# 阶段性测试与调试执行治理

## 1. 何时启用
- 任何实现、修改、修复完成后的验证。
- 任何运行时问题、阶段性联调、用户参与测试。
- 任何“只靠静态读代码无法证明正确性”的问题。

## 2. 证据优先
- 不得声称“已验证”“已测试”“已修复”，除非已经执行验证动作并保留证据。
- 每轮测试都必须至少保留以下一种证据：截图、日志、结构化结果、marker、输入输出快照。
- 结论必须能回溯到文件路径、命令输出或结构化证据，不接受“看起来没问题”。

## 3. Agent 主导
- 默认由 Agent 发起并执行大部分测试动作，包括运行命令、读取 run 目录、检查错误文件、汇总结论。
- 用户只保留必要确认和现场不可替代动作；不要把可由 Agent 直接完成的重复测试步骤转交给用户。
- 若必须用户操作，必须给出逐步指令、预期现象和需要回传的证据。

## 4. 统一循环
- 每轮开始前先明确：
  - 本轮目标
  - 验收信号
  - 失败信号
- 每轮结束后必须输出：
  - 实际结果
  - 证据路径
  - 当前结论
  - 下一步最小动作

## 5. 停止猜测
- 连续两轮无法定位根因时，禁止继续猜测。
- 必须转入增加可观测性：phase marker、结构化日志、快照、最小复现样例。
- 单个问题最多三轮“修复后重测”；达到上限仍未闭环时，必须拆分任务、记录阻塞或明确推迟。

## 6. 函数边界与可调试性
- 先按函数或模块边界定位问题：输入、自动化、截图、OCR、归一化、规则、落盘、UI。
- 优先通过小函数、纯函数和副作用边界缩小问题面。
- 不在一个大函数里混合 UI、I/O、业务判断和错误处理后再调试。

## 7. 补层不换底
- 新增 `rule`、`skill` 或局部执行流程时，只能补充漏洞、细化流程、提供执行 SOP，不得替代既有上位约束。
- `alwaysApply` 的项目护栏、证据规则、质量门禁和顶层文档基线优先于后加的 `skill` 或局部流程。
- 若出现冲突，优先遵循：顶层文档与已确认需求 > `alwaysApply` 规则 > 阶段治理规则 > 按需调用的 `skill`。

## 8. 当前项目落地要求
- 本项目阶段性测试/调试执行以 `docs/05-阶段性测试与调试执行规范.md` 为主文档。
- 涉及 Testlab、抓屏、OCR、字段抽取、规则比对、告警展示时，必须遵循：
  - 证据链完整
  - 错误信息可行动
  - 停止猜测后转插桩
  - 用户参与最小化
