# Examples

本文件用于配合 `create-nsis-frontend-patch-installer` Skill 使用，帮助执行者快速判断：

- 什么情况应该用前端补丁包
- 什么情况不应该用前端补丁包
- 输出结果应该长什么样

## 示例 1：纯前端 UI 热修

### 用户请求
```text
已安装电脑不用重装，帮我做一个只更新页面的补丁包。
```

### 适配判断
- 前端页面有修改
- 后端接口未改
- 数据结构未改
- 桌面壳未改
- 运行时未改

### 结论
应调用 `create-nsis-frontend-patch-installer`

### 期望行为
- 自动识别旧安装目录
- 升级前停桌面壳和后台
- 备份旧版 `app/frontend`
- 仅覆盖 `app/frontend`
- 保留 `data/`

## 示例 2：安装版白屏修复

### 用户请求
```text
安装版启动白屏，已经确认只需要调整前端路由和静态资源路径，帮我做一个增量更新包。
```

### 适配判断
- 问题发生在前端路由/资源层
- 后端服务无需替换
- 桌面壳可继续复用

### 结论
应调用 `create-nsis-frontend-patch-installer`

### 期望输出
```markdown
## 补丁范围
- 覆盖目录：app/frontend
- 保留目录：data/

## 输出产物
- patch installer：...
- sha256：...
```

## 示例 3：仅替换告警音和前端交互

### 用户请求
```text
这次只改了告警音资源、前端播放逻辑和一个隐藏测试入口，帮我发一个已安装机器可用的补丁安装包。
```

### 适配判断
- 改动都在前端静态资源和前端逻辑
- 无后端 schema 变化
- 无运行时变化

### 结论
应调用 `create-nsis-frontend-patch-installer`

## 示例 4：不应使用前端补丁包

### 用户请求
```text
后端接口也改了，数据库字段也有变动，但我还是想只发一个前端补丁包。
```

### 适配判断
- 后端变化存在
- 数据结构变化存在

### 结论
不应调用 `create-nsis-frontend-patch-installer`

### 正确做法
- 改走全量安装包流程
- 或设计更完整的程序层升级方案

## 示例 5：安装目标不明确

### 用户请求
```text
帮我做一个前端补丁包，用户机器上可能装过，也可能没装过。
```

### 适配判断
- 目标机器安装状态不明确
- 前端补丁包不应承担首装职责

### 结论
不能把补丁包设计成默认支持首装

### 正确做法
- 如果必须覆盖已安装机器，用补丁包
- 如果还要覆盖未安装机器，同时提供全量安装包

## 示例 6：推荐的完成输出

```markdown
## 补丁范围
- 覆盖目录：app/frontend
- 保留目录：data/

## 升级前行为
- 安装目录识别：读取历史 InstallLocation
- 停桌面壳：已执行
- 停后台：已执行 stop 脚本并做健康检查确认

## 备份策略
- 备份目录：backup/pre-frontend-update/<timestamp>/frontend

## 输出产物
- patch installer：D:\...\dist\xxx.exe
- alias：D:\...\dist\xxx-dev-x64.exe
- sha256：D:\...\dist\xxx.sha256.txt
```

## 示例 7：不合格输出

以下输出不合格：

```markdown
补丁已经做好了，给你一个 exe。
```

原因：
- 没说明覆盖范围
- 没说明是否备份
- 没说明是否停栈
- 没说明是否保留 `data/`
- 没说明产物校验文件

## 示例 8：边界判断口诀

当你犹豫是否该用这个 Skill 时，用这条判断：

```text
只改前端 -> 可以优先考虑补丁包
改了后端 -> 不要假装前端补丁包够用
改了 runtime / desktop / launcher -> 不要用前端补丁包偷懒
```
