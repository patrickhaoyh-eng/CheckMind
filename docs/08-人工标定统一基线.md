# 人工标定统一基线（长期基线）

## 1. 状态
- 状态：Accepted
- 日期：2026-07-09
- 适用范围：当前固定工位的 Testlab 自动化执行链路
- 文档定位：用于固定长期约束、治理边界与验收口径；允许实现路径按实际收敛调整，但不允许弱化质量门禁、证据要求与 fail-fast 原则
- 最新固化结论：
  - `Sine Setup` 页签验真签名已收敛为一次命中
  - `top_serial` 已切换为结构化 `RoiWindow + Sha256` 主口径，并在正式运行中达到 `configured=1; matched=1`
  - 运行结果定位已统一为免人工查找口径

## 2. 背景
- `Task A` 已证明：固定页签点击点、页签验真、PgUp/PgDn 翻页、逐屏取证这条链路可以跑通。
- 但后续核验证据显示：即使链路跑通，截图质量仍可能不足以支撑“无遗漏”的人工复核。
- 当前主要风险已经从“切不过去/翻不动”转为“截图框不稳定、覆盖不完整、运行时猜测导致漏截”。
- 因此，项目需要从“运行时策略驱动”切换为“人工标定驱动”的长期稳定基线。
- 最新风险补充：开发机存在混合 DPI（例如 150% 与 100%）与多屏组合，若进程 DPI 感知不正确，会产生坐标虚拟化与黑边/截偏，破坏“人工标定坐标 ↔ 实际截图坐标”的一致性。

## 3. 基线分层决策
- 长期基线分为两层，不再混写：
  - 治理层基线：人工标定、运行前统一核验、失败直接阻断、前端一键重标
  - 执行层基线：在已标定且已核验通过的位置上，使用 `PgUp/PgDn` 执行滑窗取证
- 后续所有关键动作均纳入人工标定体系，统一进入固定工位 profile 管理。
- 人工标定对象至少包括：
  - 页签点击点
  - 弹窗点击点
  - 验真 ROI / 验真签名
  - 截图 ROI / 表格取证框
  - 其他需要固定点击、固定截取、固定核验的 UI 关键区域
- 每次正式执行前，必须对全部已标定对象进行统一核验。
- 任一核验失败时，默认直接阻断，不允许带病运行。
- 前端必须提供重标入口；当核验失败时，用户可直接在前端发起重新标定。
- 允许因实际调试收敛而调整实现顺序、阶段先后与局部技术路径；但这些调整不得改变本文件定义的顶层约束、证据门禁、质量边界与非目标。
- 运行时读取 profile 时，优先使用结构化 `VerifyTargets`（如 `tab_verify`、`top_serial`）；旧字段 `VerifyRoiWindow / VerifySha256 / TopSerialVerifySha256` 仅保留兼容兜底，不再作为主口径。
- `top_serial` 不再只保存一个历史 `sha256`；应保存为结构化 `VerifyTargets.top_serial = { RoiWindow, Sha256 }`，并在 `table_scan` 截图框标定时自动同步更新，避免“写入口径”和“运行时验证口径”不一致。
- `top_serial` 的样本生成时机必须与正式运行一致：先聚焦表格、执行 `PgUp` 回顶并以 hash 稳定判定到顶，再从该稳定帧生成 `RoiWindow + Sha256`；不允许直接拿普通首帧或人工目测“像顶部”的截图作为样本。
- 对于 `1..N` 这类“变量在行序号、而不在窗口几何”的列表型入口（如 `Notch Profiles`），不按“每个序号单独人工标定”处理；长期主口径改为“固定列表布局 + 行定位规则 + 后续动作入口”。
- 列表型入口的人工标定主口径至少包括：
  - 列表容器 ROI
  - 首行基准
  - 行高 / 行步进
  - 后续动作入口（如 `Edit` 按钮）
- 列表型入口不承诺兼容用户任意调整行高、列宽或列表布局；若布局漂移导致行定位规则失效，preflight 必须直接阻断，不允许冒险执行。

## 4. 执行原则
- 坐标体系基线：以 `1920×1080（100% DPI）` 作为长期标定与截图的统一坐标系；该基线与正式部署机（单屏 1080p/100%）一致。
- 覆盖优先：截图采集阶段宁可重复，不允许遗漏。
- 采集从严：每次 `PgDn` 后立刻截一次，执行时严格按已标定截图框取证，不在运行时临时猜测截图区域。
- 执行动作保留：`PgUp/PgDn` 继续作为当前长期基线中的默认滑窗执行动作，不被废弃。
- 后处理去重：重复过滤、拼接优化、冗余裁汰全部后移到采集完成之后，不在采集阶段做激进裁汰。
- Fail-fast：标定核验失败、焦点不对、签名不命中、目标区域不可用时，立即阻断并给出可行动提示。
- 固定工位优先：当前基线服务于固定工位，不以“跨电脑自适应”为目标。
- 对于可调整大小与位置的子窗口，执行前必须先强制最大化，再进入页签切换、区域截图或 `PgUp/PgDn` 滑窗取证；不得在可变窗口尺寸下直接复用固定坐标。
- 子窗口关闭动作允许在“最大化后右上角关闭按钮坐标稳定”的前提下复用统一关闭坐标；若窗口未最大化或关闭按钮位置不稳定，不得复用。
- 已经跑通并固化的 `Channel Setup` / `Sine Setup` 必须纳入同一套抓取通用范式与差异化配置口径，但后续扩展遵循“补层不换底”，不得为统一而破坏现有稳定链路。
- 当新页面与既有页面存在真实交互差异时，优先拆分为“统一主链 + 差异前置动作”两层，而不是为了复用把全部流程硬揉成一套。
- 当前已确认的拆分边界是：
  - 统一主链：切页并验真、表格入口定位、`PgUp` 回顶稳定判定、`PgDn` 翻页、变化检测、去重拼接、结构化证据落盘
  - 差异前置层：进入可翻页状态、激活目标区域、获取可发键焦点
- `Notch Profile` 与 `Sine Setup` / `Channel Setup` 的当前主要差异，被明确收敛在“进入可翻页状态”这一层；在该差异未被单独封装前，不允许直接改写已稳定的统一主链。
- 本轮开发与调试默认以“先回归 `Sine Setup` / `Channel Setup`，再接 `Notch Profile`”为顺序；若统一主链回归未通过，不得继续向 `Notch Profile` 扩面。

## 5. 固定工位门禁（屏规格）
- 门禁目标：确保“人工标定坐标 ↔ 运行时截图坐标”严格一致，避免混合 DPI/多屏导致坐标虚拟化。
- 判定条件（按屏规格，不按第几屏编号）：
  - Testlab 窗口所在显示器分辨率必须为 `1920×1080`
  - Windows 缩放必须为 `100%`
- 不符合处理：
  - 正式执行前直接阻断（fail-fast），不允许带病运行
  - 退出码固定为 4（门禁失败阻断）
  - 输出 `workstation_compliance.json`（包含不符合项、期望/实际、可行动建议）
- DPI 要求：进程需启用 Per-Monitor V2 DPI 感知，避免跨屏 DPI 虚拟化造成 `2880×1620` 等“看似放大”的坐标现象。

## 6. 运行结果定位（免人工查找）
- 运行结果目录以 `%TEMP%\checkmind_testlab_last_run.txt` 为唯一事实源，脚本与排障均以该路径为准，不要求人工比对“最新文件夹”。
- 默认脚本会自动打开本次运行的 evidence 目录（若存在），并将 `last_run` 写入剪贴板，现场无需手工抄路径。

## 7. 替代与废弃
- 已验证有效的旧执行动作保留：
  - `PgUp/PgDn` 保留为执行层基线
  - “每次 `PgDn` 后立刻截一屏”保留为执行层证据节奏
- 明确废弃以下旧治理思路，不再作为长期主方案：
  - 运行时动态猜测截图框
  - 依赖 OCR 决定截图位置或覆盖完整性
  - 采集阶段优先减图、压图数量，再去赌不漏截

## 8. 成功标准
- 正式执行前可以统一核验全部人工标定对象，并输出结构化核验结果。
- 任一标定对象异常时，系统直接阻断，并给出“重新标定”入口。
- `Channel Setup`、`Sine Setup` 的截图证据按固定截图框采集后，人工能确认“不漏关键行”。
- 后续新增页面不再单独发明新策略，而是沿用同一套“标定 -> 核验 -> 执行 -> 后处理”基线。
- 当统一主链发生函数级收口或重构时，必须先以 `Channel Setup` / `Sine Setup` 回归结果证明“旧页面未退化”，再允许将该主链接入 `Notch Profile` 等新增页面。

## 9. 实施阶段
- Phase 1：把 `Channel Setup` 与 `Sine Setup` 的截图框纳入人工标定与运行前核验。
- Phase 2：把弹窗点击点、更多页面截图框、更多验真框纳入统一 profile。
- Phase 3：将前端重标能力做成正式入口，并收敛旧的临时脚本/临时基线。
