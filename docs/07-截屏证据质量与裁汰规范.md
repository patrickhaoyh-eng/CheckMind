# 截屏证据质量与裁汰规范（Testlab）

## 目标
- 支持后续“核验比对”：用户能快速确认是否截错、漏截、截屏信息是否完整。
- 支持开发调试：保留必要的失败证据与诊断线索，但不与“核验证据”混在一起。
- 固化目录与命名：脚本/代码升级后，证据组织方式保持稳定可预期。

## 目录结构（每个 run）
- `screenshots/evidence/`：用于后续核验比对与审查的证据图（默认主目录）
- `screenshots/final_compare/`：用于最终核验比对的正式截图集；只放被正式纳入口径的少量截图
- `screenshots/debug/`：用于开发调试/门禁诊断/过程定位的辅助图

## 证据图分层定义

### A. 核验证据（evidence）
核验证据应满足：能回答“有没有漏屏/截错页/截错表/每屏信息是否可读且包含关键上下文”。

**Run 级（每次执行）**
- `testlab_0_before.png`
- `testlab_1_maximized.png`

**页签切换级（每个 tab）**
- `testlab_tabs_{index}_1_after.png`

**表格入口级（每个 table）**
- `testlab_tabs_{index}_{tableKey}_table_entry.png`（表格 ROI 定位证据）

**逐屏取证级（每个 table 的每个 chunk）**
- `testlab_table_{table}_v_{chunk}.png`（表格 ROI 的逐屏证据）
- `testlab_table_{table}_stitched.png`（逐屏拼接证据）

### A+. 最终核验比对集（final_compare）
最终核验比对集用于回答“本轮 run 最后到底该看哪些图”，它不是全量证据目录，而是从 `evidence/` 中再筛出正式核验主入口。

- 当前正式核验比对集固定收以下截图：
  - `testlab_table_channelparameterstable_stitched.png`
  - `testlab_table_channelsetuptable_stitched.png`
  - `testlab_table_notchprofiletable#1_stitched.png` 到 `testlab_table_notchprofiletable#N_stitched.png`
  - `testlab_table_profileeditortablescan_stitched.png`
  - `testlab_tabs_01_advancedcontrolsetup_measurements.png`
  - `testlab_tabs_01_advancedcontrolsetup_safety.png`
  - `testlab_tabs_01_advancedcontrolsetup_throughputrecording.png`
  - `testlab_tabs_01_sinesetup_controlpanel.png`
- 这些图片应单独落到 `screenshots/final_compare/`，供最终核验直接使用。
- `evidence/` 仍保留原始证据图，`final_compare/` 只是正式核验视角下的精简投影，不替代原始留档。

### B. 调试辅助（debug）
调试辅助用于回答“为什么没切页/为什么验真不命中/为什么回顶不稳定/为什么判变化失败”，不作为核验比对主入口。

- 页签点击/探针/搜索 ROI
- profile_signature mismatch 的签名 ROI 图
- 表格扫描的 `window_v_*`、`serial_v_*`、回顶前后图、回顶签名验证图
- 翻页变化检测的 after/serial_after/fail 过程图
- 标定（calibrate）过程截图与 ROI

## 结构化证据（非图片）
核验比对与开发调试均依赖以下结构化证据文件（不分目录）：
- `testlab_phases.log`（阶段日志）
- `coverage.json`（表格扫描结果摘要）
- `table_scroll_events_*.json`（滚动事件与变化判定）
- `table_evidence_*.json`（拼接/去重一致性报告：unique chunk 数、变更事件数、是否一致）
- `tab_click_gate_*.json`（门禁失败报告，若有）

## 截屏质量最低标准（核验视角）
- 表格 ROI 图必须清晰可读：列标题/关键行/滚动位置线索至少具备其一（以“能人工核对字段”为准）。
- 同一张表的逐屏证据必须能证明覆盖推进：相邻 chunk 至少存在可观察差异（与 `table_scroll_events_*.json` 的 `Changed=true` 对齐）。
- `stitched.png` 必须能反映真实拼接顺序：与 `UniqueChunkCount` 一致，不得出现明显错序或重复拼接导致的伪覆盖。

## 拼接与裁汰规则（代码边界）
- 去重键固定为 `SerialSha256`，不是整张 ROI 图的 `Sha256`。
- `stitched.png` 只允许由“按原始出现顺序保留的首个 unique chunk”拼接生成，不允许打乱顺序重排。
- 当表格扫描从顶部开始时，理论 `UniqueChunkCount = Changed=true 的事件数 + 1`；若不成立，应视为 evidence 口径异常，需要回看 `table_scroll_events_*.json` 与 `table_evidence_*.json`。
- `table_evidence_*.json` 应至少包含：
  - `ChunkCount`
  - `UniqueChunkCount`
  - `ChangedEventCount`
  - `ExpectedUniqueChunkCount`
  - `DedupKey`
  - `IsConsistent`
- 裁汰发生在采集完成之后；采集阶段仍坚持“覆盖优先、宁可重复不允许遗漏”。

## 视觉验收最小标准
- `stitched.png` 中相邻段落应能与对应的 `testlab_table_*_v_*.png` 一一回溯，不允许出现无法追溯来源的拼接片段。
- 若 `table_evidence_*.json.IsConsistent = true`，但人工仍观察到明显错序、伪覆盖、关键列缺失或跨段跳变异常，应判为“结构一致但视觉不通过”，继续保留为待修问题，不得用结构化数量对齐替代人工验收。
- 当人工视觉验收与结构化报告冲突时，优先保留冲突现场截图、run 目录与结构化证据，不允许直接覆盖旧 run。

## 验收标准
- 新 run 下，`screenshots/` 必须包含 `evidence/`、`final_compare/` 与 `debug/` 三个子目录。
- `evidence/` 中能独立完成“截错/漏截/关键证据缺失”的人工核验。
- `final_compare/` 中只保留正式核验比对所需截图，不混入 chunk/debug 图。
- `debug/` 中仍能定位典型失败：profile_signature mismatch、回顶不稳定、翻页无变化。
- `table_evidence_*.json` 与 `coverage.json`、`table_scroll_events_*.json` 三者口径一致。
