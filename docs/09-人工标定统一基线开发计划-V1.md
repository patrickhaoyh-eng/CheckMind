# 人工标定统一基线开发计划 V1

## 1. 目标
- 在不改变“`PgUp/PgDn` 作为执行层动作”这一事实的前提下，把项目切换到“人工标定驱动”的长期基线。
- 本计划只定义开发顺序、验收口径和迁移边界，不直接代表代码已完成。
- 当前进展更新：
  - `Channel Setup` / `Sine Setup` 已按固定截图框稳定产出 evidence
  - 固定工位门禁（按屏规格）已接入并通过实机验证
  - `Sine Setup` 页签验真签名与 `top_serial` 已完成主链路收口

## 2. 计划范围
- 纳入本轮：
  - profile 扩展：支持截图框、弹窗点击点、更多验真框
  - 运行前统一核验
  - 固定工位门禁（按屏规格 `1920×1080/100%`，并启用 Per-Monitor V2 DPI）
  - 前端重标入口
  - `Channel Setup` / `Sine Setup` 截图框接入
  - 旧动态截图区域策略的迁出计划
- 不纳入本轮：
  - 跨电脑/跨分辨率自适应
  - OCR 回归主链路
  - 采集阶段的智能减图优化

## 3. 设计分层

### 3.1 治理层
- 标定对象定义
- profile 持久化
- 运行前统一核验
- 核验失败阻断
- 前端一键重标

### 3.2 执行层
- 在核验通过的前提下执行：
  - 页签点击
  - 弹窗点击
  - 验真截图
  - 表格截图
  - `PgUp/PgDn` 滑窗取证

## 4. 分阶段实施

### Phase 1：数据模型与配置基线
**目标**
- 让 profile 能表达“人工标定统一基线”所需对象。

**任务**
- 定义统一标定对象模型：
  - `ClickPoint`
  - `CaptureRoi`
  - `VerifyRoi`
  - `DialogActionPoint`
- 扩展当前固定工位 profile 文件结构
- 为 `Channel Setup`、`Sine Setup` 增加截图框配置位
- 为后续弹窗点击点预留结构

**验收标准**
- profile 可同时保存页签点击点、截图框、验真框
- 旧配置可迁移或兼容读取
- 结构命名足够稳定，避免后续再改一轮

### Phase 2：运行前统一核验
**目标**
- 在正式执行前统一验证所有已标定对象是否可用。

**任务**
- 增加统一核验入口
- 逐项核验：
  - 固定工位门禁（按屏规格，不按第几屏编号）：
    - Testlab 窗口所在显示器分辨率必须为 `1920×1080`
    - Windows 缩放必须为 `100%`
    - 进程启用 Per-Monitor V2 DPI 感知，避免混合 DPI 造成坐标虚拟化
  - 页签点击点
  - 验真 ROI / 签名
  - 截图 ROI 是否在有效窗口范围内
  - 必要时核验弹窗点击点
- 输出结构化核验结果文件
- 任一失败直接阻断执行

**验收标准**
- 核验结果可以结构化落盘
- 执行前可以看到“通过/失败/失败项”
- 失败时不进入正式抓取
- 固定工位门禁失败时有可行动建议（移动窗口到 1080p/100% 显示器、调整缩放/分辨率等）

### Phase 3：前端重标入口
**目标**
- 把“重新标定”从脚本式补丁变成正式前端能力。

**任务**
- 在前端增加“统一核验”入口
- 在前端增加“重新标定”入口
- 支持按对象类型重标：
  - 页签点击点
  - 截图框
  - 验真框
  - 弹窗点击点
- 核验失败后支持直接跳转重标

**验收标准**
- 用户不需要改脚本即可完成核验与重标
- 重标完成后 profile 能立即生效
- 核验失败提示可行动

### Phase 4：执行链路接入
**目标**
- 让执行层全部切换到“按标定对象执行”。

**任务**
- 切页动作改为只读取标定点击点
- 验真改为只读取标定验真 ROI / 签名
- `Channel Setup` / `Sine Setup` 截图改为只读取标定截图框
- 保留 `PgUp/PgDn` 作为滑窗执行动作
- 保留“每次 `PgDn` 后立刻截一屏”

**验收标准**
- 不再依赖运行时动态猜截图框
- 证据图来源可追溯到具体标定对象
- 两个页面都能按固定截图框稳定产出 evidence
- 已补充收口口径：
  - 运行时优先读取结构化 `VerifyTargets`
  - `Sine Setup` 页签验真签名应稳定一次命中
  - `top_serial` 样本必须由“聚焦表格 + PgUp 回顶稳定后”的同口径帧生成

### Phase 5：旧治理基线迁出
**目标**
- 删除或封存不再使用的旧路线，避免后续维护分叉。

**任务**
- 标记并移除动态截图区域推导逻辑
- 标记并移除与截图位置判定有关的 OCR 旧路径
- 清理只服务旧治理基线的环境变量与注释
- 更新脚本与文档，统一指向新基线

**验收标准**
- 文档、脚本、实现不再出现互相冲突的基线表述
- 新同事只看当前文档即可理解主路径

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| profile 结构扩展破坏旧读取 | 高 | 先做兼容读取，再逐步迁移 |
| 前端重标入口与现有脚本路径并存导致混乱 | 高 | 先定义单一入口，再逐步收敛脚本用途 |
| 截图框人工标定后仍漏行 | 高 | 坚持覆盖优先，保留 `PgDn` 每次截屏与后处理去重 |
| 旧动态逻辑残留导致实际运行混用新旧基线 | 高 | 在 Phase 5 做显式迁出清理 |

## 6. 推荐实施顺序
1. Phase 1：profile 扩展
2. Phase 2：统一核验
3. Phase 3：前端重标入口
4. Phase 4：执行链路接入
5. Phase 5：旧治理基线迁出

## 7. 检查点

### Checkpoint A
- 完成 Phase 1 + Phase 2 后
- 验证：不改执行逻辑，也能先拿到统一核验结果与阻断能力

### Checkpoint B
- 完成 Phase 3 后
- 验证：前端可独立完成核验与重标

### Checkpoint C
- 完成 Phase 4 后
- 验证：`Channel Setup` / `Sine Setup` 可按固定截图框 + `PgUp/PgDn` 稳定产出 evidence

### Checkpoint D
- 完成 Phase 5 后
- 验证：代码与文档只保留一套长期主路径
