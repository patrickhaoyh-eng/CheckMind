## 1.4 当前推荐拆分边界
- 统一主链函数优先拆分为以下标准步骤：
  1. 切页并验真
  2. 解析页面表入口与取证 ROI
  3. 进入表格扫描前的 `PgUp` 回顶稳定判定
  4. `PgDn` 翻页与变化检测
  5. unique chunk 去重与 stitched 拼接
  6. 结构化证据与结果落盘
- 差异前置层优先拆分为以下独立步骤：
  1. 进入目标页面后的前置激活动作
  2. 获取可翻页焦点 / 可发键状态
  3. 必要的额外交互准备（仅对差异页面开放）
- 当前明确要求：后续若要接入 `Notch Profile`，只能扩展差异前置层，不得先改写统一主链的既有成功口径。

## 1.1 不变约束
- 不改变顶层质量约束：证据优先、可追溯、fail-fast、停止猜测后转插桩。
- 不改变治理边界：人工标定、统一核验、固定工位门禁、结构化 profile 主口径。
- 不改变非目标：不扩展自动改参/自动点击修改/自动阻断试验，不回退到 OCR 主链路，不做跨电脑自适应承诺。

## 1.2 可调整项
- 可根据实机问题收敛情况调整阶段先后顺序。
- 可优先处理影响主链路真实性、证据完整性和口径一致性的收口项。
- 可在保留兼容兜底的前提下，逐步收敛旧字段、旧脚本和旧治理路径。

## 4. 分阶段实施

### Phase 1：数据模型与配置基线
**目标**
- 让 profile 能表达“人工标定统一基线”所需对象。

**任务**
- 定义统一标定对象模型：
  - `ClickPoint`
  - `CaptureRoi`
  - `VerifyRoi`
  - `DialogActionPoint`
- 扩展当前固定工位 profile 文件结构
- 为 `Channel Setup`、`Sine Setup` 增加截图框配置位
- 为后续弹窗点击点预留结构

**验收标准**
- profile 可同时保存页签点击点、截图框、验真框
- 旧配置可迁移或兼容读取
- 结构命名足够稳定，避免后续再改一轮

### Phase 2：运行前统一核验
**目标**
- 在正式执行前统一验证所有已标定对象是否可用。

**任务**
- 增加统一核验入口
- 逐项核验：
  - 固定工位门禁（按屏规格，不按第几屏编号）：
    - Testlab 窗口所在显示器分辨率必须为 `1920×1080`
    - Windows 缩放必须为 `100%`
    - 进程启用 Per-Monitor V2 DPI 感知，避免混合 DPI 造成坐标虚拟化
  - 页签点击点
  - 验真 ROI / 签名
  - 截图 ROI 是否在有效窗口范围内
  - 必要时核验弹窗点击点
- 输出结构化核验结果文件
- 任一失败直接阻断执行

**验收标准**
- 核验结果可以结构化落盘
- 执行前可以看到“通过/失败/失败项”
- 失败时不进入正式抓取
- 固定工位门禁失败时有可行动建议（移动窗口到 1080p/100% 显示器、调整缩放/分辨率等）

### Phase 3：前端重标入口
**目标**
- 把“重新标定”从脚本式补丁变成正式前端能力。

**任务**
- 在前端增加“统一核验”入口
- 在前端增加“重新标定”入口
- 支持按对象类型重标：
  - 页签点击点
  - 截图框
  - 验真框
  - 弹窗点击点
- 核验失败后支持直接跳转重标

**验收标准**
- 用户不需要改脚本即可完成核验与重标
- 重标完成后 profile 能立即生效
- 核验失败提示可行动

### Phase 4：执行链路接入
**目标**
- 让执行层全部切换到“按标定对象执行”。

**任务**
- 切页动作改为只读取标定点击点
- 验真改为只读取标定验真 ROI / 签名
- `Channel Setup` / `Sine Setup` 截图改为只读取标定截图框
- 保留 `PgUp/PgDn` 作为滑窗执行动作
- 保留“每次 `PgDn` 后立刻截一屏”

**验收标准**
- 不再依赖运行时动态猜截图框
- 证据图来源可追溯到具体标定对象
- 两个页面都能按固定截图框稳定产出 evidence
- 已补充收口口径：
  - 运行时优先读取结构化 `VerifyTargets`
  - `Sine Setup` 页签验真签名应稳定一次命中
  - `top_serial` 样本必须由“聚焦表格 + PgUp 回顶稳定后”的同口径帧生成

### Phase 5：旧治理基线迁出
**目标**
- 删除或封存不再使用的旧路线，避免后续维护分叉。

**任务**
- 标记并移除动态截图区域推导逻辑
- 标记并移除与截图位置判定有关的 OCR 旧路径
- 清理只服务旧治理基线的环境变量与注释
- 更新脚本与文档，统一指向新基线

**验收标准**
- 文档、脚本、实现不再出现互相冲突的基线表述
- 新同事只看当前文档即可理解主路径

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| profile 结构扩展破坏旧读取 | 高 | 先做兼容读取，再逐步迁移 |
| 前端重标入口与现有脚本路径并存导致混乱 | 高 | 先定义单一入口，再逐步收敛脚本用途 |
| 截图框人工标定后仍漏行 | 高 | 坚持覆盖优先，保留 `PgDn` 每次截屏与后处理去重 |
| 旧动态逻辑残留导致实际运行混用新旧基线 | 高 | 在 Phase 5 做显式迁出清理 |

- `Notch Profiles` 的 `1..N` 列表是后续待抓取素材的入口，不按“每个序号单独人工标定”处理。

- 为兼容现有调试/脚本入口，执行层继续保留 `CHECKMIND_NOTCH_PROFILE_INDEX` / `CHECKMIND_NOTCH_PROFILE_INDEXES`；若显式提供序号列表，则优先于 `notchProfileCount` / `CHECKMIND_NOTCH_PROFILE_COUNT`。
- 前后端联调用字段与校验口径已单独固化到 `docs/10-正弦任务接口约定-V1.md`，后续前端接入按该文档执行，不再依赖会话记录。
