## 5. 每轮必须收集的证据

每次回归至少收集以下 4 类证据，禁止只看终端现象就下结论。

### 5.1 终端证据

- 实际执行命令
- 终端末尾输出
- `last_run=...`
- `formal_verification_summary_file=...`
- 成功时还要记录：
  - `[CheckMind] formal_verification_summary_begin`
  - `task_result_status=completed`
  - `[CheckMind] formal_verification=passed`
- 若为 mismatch，还要记录：
  - `[CheckMind] formal_verification=failed`
  - `failed_objects=...`
  - `reason_message[...] = ...`
  - `notch_profile_count_mismatch=...`

### 5.2 结构化结果

- `results.json`
- `results.json.formalVerificationSummary`
  - 用于提供对象级正式核验摘要，快速回答每个对象是否通过，而无需先深入 `testlab_run.json`
  - 顶层补统一汇总：
    - `allVerified`
    - `failedObjects`
    - `failedReasons`
  - `failedReasons` 字段职责拆分：
    - `reasonCodes`：机器可读短码，供脚本、前端、报告做稳定分支判断
    - `reasonMessages`：面向人类的短句，供弹窗、报告、前端直接展示
- `formal_verification_summary.json`
  - 位于每个正式业务 run 根目录
  - 只复用 `results.json.formalVerificationSummary`、`taskResult`、`alerts` 与 `finalCompareDirectory`，不新增新的判定来源
  - 供批量回归、报告聚合和静默模式直接消费，无需再解析终端历史
- `artifacts/probe-runs/_reports/formal_verification_batch_summary.json`
- `artifacts/probe-runs/_reports/formal_verification_batch_summary.md`
  - 由 `scripts/export_formal_verification_batch_summary.ps1` 离线生成
  - 默认汇总全部 run；若传入 `-TaskType` / `-Limit`，输出文件名会自动带筛选后缀
  - 当前已验证筛选产物：
    - `artifacts/probe-runs/_reports/formal_verification_batch_summary_tasktype_sine_latest_2.json`
    - `artifacts/probe-runs/_reports/formal_verification_batch_summary_tasktype_sine_latest_2.md`
- `testlab_run.json`
- `testlab_phases.log`
- `testlab_run.json.FinalCompareArtifacts`
  - 用于明确本轮最终核验截图集，不再让人工从全量 `evidence/` 中手工筛图
- `testlab_run.json.ChannelSetupFormalResult`
  - 用于固定 `Channel Setup` 主表扫描的正式核验字段
- `testlab_run.json.SineSetupFormalResult`

### 5.5 失败短码字典

`formalVerificationSummary.failedReasons` 只输出机器可读短码，不复制长文本。当前正式字典如下：

- 通用短码：
  - `UNIQUE_CHUNKS_EMPTY`
    - 含义：预期应形成去重后的表格块，但最终唯一块数量为 0
  - `FINAL_COMPARE_SCREENSHOT_MISSING`
    - 含义：对象级最终核验截图缺失
  - `CHILD_WINDOW_NOT_CLOSED`
    - 含义：子窗口扫描完成后未正常关闭
  - `NOT_RETURNED_TO_PARENT`
    - 含义：子窗口关闭后未正常回到父窗口
  - `WINDOW_ERROR`
    - 含义：对象级正式结果中存在窗口错误信息

- `channelSetup`：
  - `UNIQUE_CHUNKS_EMPTY`
  - `FINAL_COMPARE_SCREENSHOT_MISSING`

- `sineSetup`：
  - `CHANNEL_PARAMETERS_UNIQUE_CHUNKS_EMPTY`
    - 含义：`Channel Parameters Table` 未形成有效唯一块
  - `CHANNEL_PARAMETERS_FINAL_COMPARE_MISSING`
    - 含义：`Channel Parameters Table` 的最终核验拼接图缺失
  - `CONTROL_PANEL_FINAL_COMPARE_MISSING`
    - 含义：`Control Panel` 的最终核验截图缺失

- `advancedControlSetup`：
  - `FINAL_COMPARE_SET_INCOMPLETE`
    - 含义：`measurements / safety / throughput recording` 三张正式核验截图至少缺一张
  - `CHILD_WINDOW_NOT_CLOSED`
  - `NOT_RETURNED_TO_PARENT`
  - `WINDOW_ERROR`

- `notchProfiles`：
  - `COUNT_MISMATCH`
    - 含义：请求数量与实际完成数量不一致
  - `ROW_FLOW_NOT_VERIFIED`
    - 含义：至少一条 `Notch Profile` 子窗口未形成正式流程通过

- `profileEditor`：
  - `WINDOW_NOT_OPENED`
    - 含义：未真正进入 `Profile Editor` 子窗口
  - `UNIQUE_CHUNKS_EMPTY`
  - `FINAL_COMPARE_SCREENSHOT_MISSING`
  - `CHILD_WINDOW_NOT_CLOSED`
  - `NOT_RETURNED_TO_PARENT`
  - `WINDOW_ERROR`

### 5.6 失败展示文案口径

- `formalVerificationSummary.failedReasons.reasonMessages` 由统一字典生成，不允许前端、脚本、报告各自硬编码一套文案。
- 当前口径：
  - 机器分支判断一律使用 `reasonCodes`
  - 面向用户的直接展示优先使用 `reasonMessages`
  - 若后续需要多语言，先保留 `reasonCodes` 不变，再扩展示文案层，不改短码语义

## 7. 失败判据

出现以下任一情况，当前回归判失败：

- 没有 `last_run`
- `results.json` 未落盘
- `testlab_run.json` 未落盘
- `Define notch profiles` 因纯哈希漂移被提前阻断
- `Notch Profile` 子窗口未打开但结果被误写为成功
- mismatch 落到错误序号
- 入口口径与提示文案不一致

## 8. 推荐执行顺序

建议按以下顺序执行，避免一上来就混入多种失败模式：

1. 主口径成功：`-NotchProfileCount 3`
2. 主口径 mismatch：`-NotchProfileCount 4`
3. 显式索引成功：`-NotchProfileIndexes "1,2,3"`
4. 显式索引 mismatch：`-NotchProfileIndexes "1,2,3,4"`

理由：

- 先证明真实主口径成功
- 再证明主口径 warning
- 然后再验证调试兼容入口是否与主口径行为一致

## 9. 最小执行 SOP

每轮都按以下固定动作执行：

1. 运行标准命令
2. 记录终端末尾 `last_run=...`
3. 读取对应 run 目录下的：
   - `results.json`
   - `testlab_run.json`
   - `testlab_phases.log`
4. 对照本计划第 6 节判定通过/失败
5. 若失败，先写清：
   - 实际结果
   - 证据路径
   - 当前结论
   - 下一步最小动作
