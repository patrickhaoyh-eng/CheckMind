# 正弦任务级端到端回归计划

## 1. 目标

- 把当前已在专项脚本中闭环的 `Notch Profiles` 能力，放回“任务级执行链路”中做一次完整回归。
- 证明当前节点不仅能通过专项 smoke，还能在真实任务语义下稳定完成：
  - 任务输入
  - preflight 门禁
  - `Channel Setup`
  - `Sine Setup`
  - `Define notch profiles`
  - `Notch Profile` 子窗口扫描
  - 运行结果落盘

## 2. 范围

- 任务类型：`sine`
- 当前主口径：`notchProfileCount`
- 对照口径：`NotchProfileIndexes`
- 当前只覆盖已形成基线的 `Notch Profiles` 相关能力，不扩展随机任务，不新增前端页面验证。

## 3. 前置条件

- Testlab 工位与当前 `workstation_profile.json` 保持一致。
- `artifacts/probe-runs/_config/workstation_profile.json` 已包含当前稳定基线。
- 当前 Git 节点已固定为：
  - commit `43eb2f2`
  - `feat: solidify testlab notch profile baseline`
- 运行前确认：
  - Testlab 已打开并可操作
  - `Sine Setup` 相关页面可正常进入
  - 若使用弹窗模式，现场允许脚本弹出确认框

## 4. 标准回归命令

### 4.1 主口径成功路径

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_testlab_with_notch_profiles.ps1 -NotchProfileCount 3
```

预期：

- `results.json.status = completed`
- `completedNotchProfileCount = 3`
- `alerts = []`

### 4.2 主口径 mismatch 路径

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_testlab_with_notch_profiles.ps1 -NotchProfileCount 4
```

预期：

- `results.json.status = completed_with_warning`
- `resultCode = NOTCH_PROFILE_COUNT_MISMATCH`
- `failedRow = 4`

### 4.3 显式索引成功路径

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_testlab_with_notch_profiles.ps1 -NotchProfileIndexes "1,2,3"
```

预期：

- `results.json.status = completed`
- `resultCode = null`
- `NotchProfileCountMismatch = null`

### 4.4 显式索引 mismatch 路径

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run_testlab_with_notch_profiles.ps1 -NotchProfileIndexes "1,2,3,4"
```

预期：

- 终端输出 `notch_profile_count_mismatch=requested:4;completed:3;failedRow:4`
- `results.json.resultCode = NOTCH_PROFILE_COUNT_MISMATCH`
- `message` 使用 `Notch Profile Indexes` 口径

## 5. 每轮必须收集的证据

每次回归至少收集以下 4 类证据，禁止只看终端现象就下结论。

### 5.1 终端证据

- 实际执行命令
- 终端末尾输出
- `last_run=...`
- 若为 mismatch，还要记录 `notch_profile_count_mismatch=...`

### 5.2 结构化结果

- `results.json`
- `testlab_run.json`
- `testlab_phases.log`
- 必要时补充：
  - `preflight_calibration_report.json`
  - `define_notch_profiles_layout_hash_warning.json`

### 5.3 页面/子窗口行为证据

- `Channel Setup` 是否完成切页/抓取
- `Sine Setup` 是否恢复到 `Channel safety parameters`
- `Define notch profiles` 是否通过第二刀门禁
- `Notch Profile` 子窗口是否按序号逐个打开、扫描、关闭

### 5.4 结论证据

- 成功路径：
  - `completedNotchProfileCount` 与预期一致
- mismatch 路径：
  - `completedNotchProfileCount`
  - `requestedNotchProfileCount`
  - `failedRow`
  - `UserMessage`

## 6. 通过判据

### 6.1 任务级通过

满足以下条件才算“任务级通过”：

- `results.json` 已正常落盘
- `testlab_run.json` 已正常落盘
- `testlab_phases.log` 已正常落盘
- `Channel Setup`、`Sine Setup`、`Notch Profiles` 三段证据连续
- 未出现“只在专项 smoke 中通过、任务级运行退化”的现象

### 6.2 新门禁通过

满足以下条件才算“第二刀门禁通过”：

- phase 中出现 `runner.notch_profiles_layout_geometry_verified`
- 没有因哈希漂移直接阻断正式链路
- 若出现哈希漂移，允许生成 `define_notch_profiles_layout_hash_warning.json`

### 6.3 mismatch 口径通过

满足以下条件才算“数量不符口径通过”：

- mismatch 精确定位到真实缺失行
- 不重复打开上一条旧内容
- 终端、`results.json`、`testlab_run.json` 三处口径一致
- 提示文案与入口口径一致：
  - 主口径：`Notch Profile Count`
  - 显式索引：`Notch Profile Indexes`

## 7. 失败判据

出现以下任一情况，当前回归判失败：

- 没有 `last_run`
- `results.json` 未落盘
- `testlab_run.json` 未落盘
- `Define notch profiles` 因纯哈希漂移被提前阻断
- `Notch Profile` 子窗口未打开但结果被误写为成功
- mismatch 落到错误序号
- 入口口径与提示文案不一致

## 8. 推荐执行顺序

建议按以下顺序执行，避免一上来就混入多种失败模式：

1. 主口径成功：`-NotchProfileCount 3`
2. 主口径 mismatch：`-NotchProfileCount 4`
3. 显式索引成功：`-NotchProfileIndexes "1,2,3"`
4. 显式索引 mismatch：`-NotchProfileIndexes "1,2,3,4"`

理由：

- 先证明真实主口径成功
- 再证明主口径 warning
- 然后再验证调试兼容入口是否与主口径行为一致

## 9. 最小执行 SOP

每轮都按以下固定动作执行：

1. 运行标准命令
2. 记录终端末尾 `last_run=...`
3. 读取对应 run 目录下的：
   - `results.json`
   - `testlab_run.json`
   - `testlab_phases.log`
4. 对照本计划第 6 节判定通过/失败
5. 若失败，先写清：
   - 实际结果
   - 证据路径
   - 当前结论
   - 下一步最小动作
